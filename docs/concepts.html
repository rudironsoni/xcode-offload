model

Apple paths stay real.

The transparent mode makes Apple expected paths real mountpoints backed by external APFS sparsebundles. Shims can help with explicit flag routing, but they are not the storage mechanism.

Storage root

Every command accepts --root PATH, or reads XCODE_STORAGE_ROOT. The root should be a mounted external volume or another writable storage location you chose explicitly.

Why APFS sparsebundles

Sparsebundles keep large developer data outside the internal disk while still giving macOS a normal APFS filesystem at the mountpoint. They also let the tool check ownership, filesystem type, and backend provenance with diskutil, mount, and hdiutil info.

Why not symlinks

CoreSimulator and simdiskimaged can reject symlinked mount parents. xcode-storage treats symlinked managed paths as invalid and refuses to proceed until they are replaced with real directories.

Launchd responsibilities

The user LaunchAgent remounts user-owned paths. The system LaunchDaemon runs a root-owned helper for /Library/Developer/CoreSimulator/* paths. daemon install and launchd install install the same system job.

Shims are optional

Shims are useful for explicit argument routing, build products, archives, and TMPDIR. They are optional when APFS mount mode is active.
