model

Apple paths stay real.

Transparent mode mounts external APFS sparsebundles at the paths Apple tools already use. Shims are optional. The mounts are the storage mechanism.

Storage root

Every command accepts --root PATH, or reads XCODE_STORAGE_ROOT. Point it at a mounted external volume or another writable location you chose.

Why APFS sparsebundles

Sparsebundles keep the heavy data off the internal disk while giving macOS a normal APFS filesystem at the mountpoint. They also give the tool something concrete to inspect with diskutil, mount, and hdiutil info.

Why not symlinks

CoreSimulator and simdiskimaged can reject symlinked mount parents. xcode-storage treats a symlinked managed path as invalid. Replace it with a real directory before mounting.

Launchd responsibilities

The user LaunchAgent remounts user-owned paths. The system LaunchDaemon runs a root-owned helper for /Library/Developer/CoreSimulator/*. daemon install and launchd install install the same system job.

xcodes compatibility

The xcodes profile keeps installed Xcode apps under /Applications/Xcodes, backed by APFS. Set XCODES_DIRECTORY to that path so xcodes installs there, then use xcode-select as usual.

Shims are optional

Shims help with explicit argument routing, build products, archives, and TMPDIR. When APFS mount mode is active, default Apple tools can use the mounted paths without shims.
