macOS developer storage

Move Xcode storage off the internal disk.

xcode-offload mounts external APFS sparsebundles at the paths Xcode and CoreSimulator already use.

Start setup Read commands

Why this exists

Xcode installs, simulator device sets, DerivedData, archives, and CoreSimulator caches can take hundreds of gigabytes. Symlinking those folders is fragile, especially around CoreSimulator images and simdiskimaged. This tool uses APFS mounts instead.

Apple paths stay normal

Tools still see paths such as ~/Library/Developer/CoreSimulator/Devices and /Library/Developer/CoreSimulator/Caches.

No symlinked mount parents

The mount manager rejects symlinked Apple paths because CoreSimulator can reject them.

Fails before it guesses

If a managed path is mounted from an unexpected backend, the tool stops and asks you to inspect it.

Repair is repeatable

LaunchAgents, LaunchDaemons, helpers, shims, and mount state are managed by commands you can run again.

Basic setup

export XCODE_OFFLOAD_ROOT="/Volumes/YourExternalVolume"

xcode-offload repair \
  --root "$XCODE_OFFLOAD_ROOT" \
  --home "$HOME" \
  --scope user \
  --install-shims \
  --dry-run

xcode-offload repair \
  --root "$XCODE_OFFLOAD_ROOT" \
  --home "$HOME" \
  --scope user \
  --install-shims \
  --load

sudo xcode-offload daemon install \
  --root "$XCODE_OFFLOAD_ROOT" \
  --home "$HOME"

xcode-offload doctor \
  --root "$XCODE_OFFLOAD_ROOT" \
  --require-shims \
  --strict

Mount mode

Use mounts when you want the default Apple paths backed directly by APFS sparsebundles: 

xcode-offload mounts install --root "$XCODE_OFFLOAD_ROOT" --home "$HOME" --scope user --load
sudo xcode-offload mounts install --root "$XCODE_OFFLOAD_ROOT" --home "$HOME" --scope system --load
xcode-offload mounts status --root "$XCODE_OFFLOAD_ROOT" --home "$HOME" --scope all
Do not symlink CoreSimulator Images or Volumes. Use mounted APFS sparsebundles. Xcode and simdiskimaged can reject symlinked mount parents.
